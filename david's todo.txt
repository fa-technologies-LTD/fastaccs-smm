P0 — MUST FIX BEFORE FULL LIVE TRAFFIC
==================================================

[#1] Client price is trusted during order creation (critical)
- Problem: order amount is derived from client-sent item.price.
- Status: DONE (server now resolves tier prices from DB and ignores client price).
- Fix plan:
  1. Ignore client price entirely.
  2. Fetch authoritative tier price from DB by categoryId.
  3. Recompute subtotal/total server-side only.
  4. Save order item unitPrice/totalPrice from DB price.
  5. Reject request if any tier price missing/invalid.

[#4] Account allocation race (TOCTOU)
- Problem: find available accounts, then update later (non-atomic).
- Status: DONE (allocation is transactional with row locks / SKIP LOCKED, including manual process path).
- Fix plan:
  1. Move allocation into a single DB transaction.
  2. Use row locking / atomic update pattern to prevent double allocation.
  3. Fail cleanly if requested quantity cannot be atomically reserved.
  4. Add concurrency test (two parallel checkouts same tier).

[#5] Wallet debit race
- Problem: balance check happens before transaction.
- Status: DONE (balance guard and decrement are now atomic in one transaction).
- Fix plan:
  1. Move balance read + sufficiency check inside transaction.
  2. Use atomic decrement guard (only update when balance >= amount).
  3. Return deterministic insufficient-balance error if guarded update fails.

[#3] Admin order PATCH mass assignment
- Problem: spreads unvalidated request body into prisma update.
- Status: DONE (strict allowlist + value validation + unknown-field rejection).
- Fix plan:
  1. Introduce explicit allowlist of editable fields.
  2. Validate each field type/value.
  3. Reject unknown keys.

[#2] Raw session token stored as session ID
- Problem: DB compromise instantly reuses active sessions.
- Status: DONE (session IDs now store sha256(token); token is hashed on lookup).
- Fix plan:
  1. Store sha256(token) as session id.
  2. Hash cookie token before DB lookup.
  3. Rotate/invalidate existing sessions during migration.

[#6] Background schedulers not reliable on serverless
- Problem: setInterval in app process is unreliable on Vercel.
- Status: DONE (moved to authenticated cron endpoints + vercel.json schedules).
- Fix plan:
  1. Move payment reconciliation to scheduled trigger (Vercel Cron or external).
  2. Move winback to scheduled trigger.
  3. Keep idempotency so repeated runs are safe.

[#9] Verification code uses Math.random
- Problem: weak entropy for security code.
- Status: DONE (uses crypto.randomInt).
- Fix plan:
  1. Replace with crypto.randomInt(100000, 1000000).

[#8] Missing rate limits for login/signup
- Problem: brute-force risk.
- Status: DONE (IP + identity rate limits added to login/signup).
- Fix plan:
  1. Add IP/email based rate limiting for login and signup.
  2. Add short lockouts/backoff after repeated failures.
  3. Keep current verification rate limits.

[#23] Password max length not enforced
- Problem: potential scrypt abuse on huge passwords.
- Status: DONE (max 72-byte cap enforced).
- Fix plan:
  1. Enforce max password length (e.g. 72 bytes or chosen cap).
  2. Return clean validation error.
